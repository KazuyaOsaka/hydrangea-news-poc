# Hydrangea — バッチプロトコル (BATCH_PROTOCOL.md)

最終更新: 2026-05-02 (F-cleanup-merge-streak 完了)

このドキュメントは、Hydrangea の全バッチで Claude Code が必ず実施すべき
共通タスクを定義する。各バッチプロンプトの末尾でこのプロトコルを参照する。

---

## 目的

Phase A.5-2 で 7 連続バッチを進めた結果、過去の決定や予定が散逸する
問題が発生した。「台本の日本語改善」「document 更新」「手動 PoC」等の
重要事項が「忘れ去られていた」ため、月 1 棚卸しでは間に合わない。

→ 都度更新を強制化する仕組みとして本プロトコルを確立する。

---

## 必須タスク (各バッチ完了時)

### Task 1: DECISION_LOG.md への決定記録

各バッチ完了時に必ず `docs/DECISION_LOG.md` に以下のエントリを追加する。

```markdown
## YYYY-MM-DD: [バッチ名] — [バッチタイトル]

### 背景
[なぜこのバッチが必要だったか]

### 議論
[どのような選択肢を検討し、なぜこの方針を採用したか]

### 決定
[実装方針の核心]

### 結果
[実装後の効果、後続バッチへの影響]

### 関連ファイル・コミット
- コミット: [ハッシュ]
- 変更: [ファイルリスト]
```

エントリは時系列順 (末尾追加) で配置する。記載粒度は既存エントリ
(F-1 / F-2 / F-13.B 等) に揃える。

### Task 2: FUTURE_WORK.md の更新

#### 2-A: 完了済みへの移動

- 本バッチで完了した項目を「完了済み」セクションに移動する
- 簡潔な完了記録 (発生バッチ / 対応内容 / 関連ファイル) を残す

#### 2-B: 新規課題の追加

本バッチで発生した新たな将来課題を「緊急度 高 / 中 / 低」のいずれかに
追加する。形式は以下:

```markdown
- **[タイトル]** (発生バッチ)
  - 背景: なぜこれが必要か
  - 対応案: どう対応するか
  - 検討時期: いつ判断するか
  - 関連ファイル: 影響を受けるファイル
```

緊急度の判断基準:
- **緊急度 高**: 次のフェーズ (次の数バッチ以内) で必ず対応すべき
- **緊急度 中**: 実運用データ収集後に判断、または別バッチで計画的に対応
- **緊急度 低**: 時間ある時に検討、Phase 完了後の整理対象

### Task 3: 完了レポートに更新内容を明記

各バッチの完了レポートに必ず以下のセクションを含める。

```markdown
## DECISION_LOG.md への追加内容
[追加したエントリの要約]

## FUTURE_WORK.md への変更内容
- 完了済みに移動した項目: [リスト]
- 緊急度 高に追加した項目: [リスト]
- 緊急度 中に追加した項目: [リスト]
- 緊急度 低に追加した項目: [リスト]
```

これらは省略不可。完了レポート提出前に必ず Task 1 / Task 2 を実施し、
その結果を Task 3 として完了レポートに反映すること。

### Task 4: docs/DISCUSSION_NOTES.md の整理

#### 4-A: 新規エントリの追加

本バッチで発生した「議論中だがまだ確定していないメモ」を
`docs/DISCUSSION_NOTES.md` の「未分類 (Active)」セクションに追加する。
形式:

```markdown
### YYYY-MM-DD: <トピック>
- **内容**: <議論内容、選択肢、論点>
- **出典**: <チャット URL / DECISION_LOG エントリ / コードベース確認等>
- **ステータス**: `Active` / `要確認` / `昇格候補(DECISION_LOG)` /
  `昇格候補(FUTURE_WORK)`
```

#### 4-B: 既存エントリの再評価

バッチ完了時に、DISCUSSION_NOTES.md の既存エントリを以下のいずれかに
振り分ける:

- **確定済み** → DECISION_LOG.md に昇格 (Task 1 のエントリとして追加し、
  DISCUSSION_NOTES.md からは削除)
- **タスク化** → FUTURE_WORK.md に昇格 (Task 2 のエントリとして追加し、
  DISCUSSION_NOTES.md からは削除)
- **アーカイブ** → 30 日以上古い + 状況変化で意味を失ったものを
  「アーカイブ」セクションに移動 (削除はしない、履歴として残す)
- **継続** → 何もせず Active のまま残す

完了レポートに「再評価結果」を記載する (どのエントリをどこに
昇格 / アーカイブしたか)。

### Task 5: docs/CURRENT_STATE.md の全置換更新

`docs/CURRENT_STATE.md` は **全置換更新** する (追記ではない)。
バッチ完了時に以下のセクションを最新化する:

1. リポジトリ状態 (main HEAD ハッシュ、直近 5 件のログ、baseline テスト数)
   ★必須: main HEAD ハッシュは `git log -1 --format=%H` で実測値を取得して反映する。
   直近 5 件ログは `git log --oneline -5` の出力を反映する。
2. 現在のフェーズ (Phase 名、進行中バッチ、次バッチ候補と推奨)
3. 直近の試運転結果サマリー (3-5 件)
4. Hydrangea コンセプト防衛機構の現状 (4+1 層の表)
5. 触ってよい / 触ってはいけない領域マップ
6. 不変原則 5 つのリマインダ
7. カズヤの直近フィードバック要点
8. 関連ドキュメントへの導線

更新後、`git diff HEAD docs/CURRENT_STATE.md` で全置換が行われたことを確認する。

## 不変原則 (Hydrangea Phase A.5-3a 時点版、F-state-protocol で正確化)

各バッチでは以下の 5 つの不変原則を遵守する:

- **不変原則 1**: `src/generation/article_writer.py` 一切変更不可
- **不変原則 2**: `src/generation/script_writer.py` の **既存ルート**
  (`write_script` / `_PROMPT_TEMPLATE` / `_build_script_from_llm`) は
  変更不可。
  - **新ルート** (`generate_script_with_analysis` /
    `ScriptWithAnalysisDraft` / `_AXIS_TO_PATTERN_HINT` /
    `_ANALYSIS_DURATION_PROFILES` / `article_text` 等) への追加・修正は OK
  - `_CHAR_BOUNDS` 等の定数調整も最小改変なら許容
  - **例外**: `configs/prompts/` 配下のプロンプトファイルは変更可能、
    主戦場は `configs/prompts/analysis/geo_lens/`
- **不変原則 3**: `src/triage/` の既存ファイル変更不可
  - **例外**: 新規ファイルの追加は可能 (例: `jp_coverage_verifier.py`)
- **不変原則 4**: `src/analysis/` 変更不可
  - **例外条項検討**: F-12-B-2 (axis 多様化) 着手時に再評価
- **不変原則 5**: 既存テスト破壊しない (baseline 1315 passed 維持)

各原則の例外は明示的に許可された場合のみ。例外を必要と判断した場合は
作業を停止し、完了レポートで「変更要望」として報告すること。

## 拡張性差し込み判断ルール (F-doc-cleanup / 2026-05-03 確立)

各バッチ着手時、Phase 正規順序の前倒しで拡張性レイヤーを実装する判断は、
以下の 3 条件を **全部** 満たす場合のみ許可する:

1. **既に実装先が存在する** — 抽象クラスだけ作る状況ではない (実装先が 0 or 1 の
   場合は前倒し却下)
2. **前倒しが将来の改修コストを実測値で削減できる根拠がある** — 推測・予感・
   「将来のため」は不可。具体的なシナリオで「今やらないとこれだけコスト増える」が
   言語化できること
3. **前倒しで「動くものを壊さない」哲学に違反しない** — 既存パイプライン無改修で
   差し込めること

1 つでも欠ければ「過剰拡張性の罠」(クラウド誤り 6) として却下し、Phase の
正規順序を守る。

### 過去の判断例

- **ChannelConfig YAML 化 (2026-05-03 判断)**: 却下 (条件 1 NG: geo_lens のみ
  実装、japan_athletes / k_pulse は未着手)。Phase 1-A の正規タイミングで実施。
- **Publisher 抽象 (2026-05-03 判断)**: 却下 (条件 1 NG: 投稿先実装そのものが
  Phase A.5-3d で初実装)。Phase A.5-3d 着手時に最初から抽象クラスとして設計する。
- **Audio/Image/Video Renderer 抽象化 (2026-05-03 判断)**: 却下 (条件 1 NG:
  実装先 1 つずつしかない)。Phase A.5-3c の各統合バッチで「抽象化 + 実装」を
  セット実施。
- **Content Format 抽象化 (2026-05-03 判断)**: 却下 (条件 1 NG: 実装先は
  Markdown 1 つのみ)。Web メディア展開時は既存 Markdown を UI に流し込むだけで足りる。

### 例外: 文書層の拡張性

本ルールは **コード層** の拡張性判断に適用する。
docs / configs / プロンプト等の文書層は「対症療法じゃなくて根本治療」哲学が
優先するため、本ルールではなく BATCH_PROTOCOL Task 4-B (DISCUSSION_NOTES 再評価) /
Task 5 (CURRENT_STATE 全置換更新) の判断軸を使う。

## バッチプロンプトテンプレートでの参照方法

各バッチプロンプトの末尾に以下のセクションを必ず含める。

```markdown
## 文書更新義務 (BATCH_PROTOCOL に従う)

本バッチ完了時に docs/BATCH_PROTOCOL.md に従い、以下を必ず実施する:

1. **Task 1**: docs/DECISION_LOG.md に本バッチの決定エントリを追加
2. **Task 2**: docs/FUTURE_WORK.md の完了済み移動 + 新規課題追加
3. **Task 3**: 完了レポートに上記更新内容を明記
4. **Task 4**: docs/DISCUSSION_NOTES.md の整理
   (4-A 新規追加 + 4-B 既存再評価)
5. **Task 5**: docs/CURRENT_STATE.md の全置換更新

これらは省略不可。完了レポート提出前に必ず実施すること。
```

---

## メンテナンスルール

- 不変原則の追加・変更は本ドキュメントを更新する
- バッチプロトコル自体の変更も DECISION_LOG.md に記録する
- 形骸化を防ぐため、月 1 回のレビュー対象とする
  (`docs/FUTURE_WORK.md` の月次レビュートリガーと同期)
