# Hydrangea — バッチプロトコル (BATCH_PROTOCOL.md)

最終更新: 2026-05-01

このドキュメントは、Hydrangea の全バッチで Claude Code が必ず実施すべき
共通タスクを定義する。各バッチプロンプトの末尾でこのプロトコルを参照する。

---

## 目的

Phase A.5-2 で 7 連続バッチを進めた結果、過去の決定や予定が散逸する
問題が発生した。「台本の日本語改善」「document 更新」「手動 PoC」等の
重要事項が「忘れ去られていた」ため、月 1 棚卸しでは間に合わない。

→ 都度更新を強制化する仕組みとして本プロトコルを確立する。

---

## 必須タスク (各バッチ完了時)

### Task 1: DECISION_LOG.md への決定記録

各バッチ完了時に必ず `docs/DECISION_LOG.md` に以下のエントリを追加する。

```markdown
## YYYY-MM-DD: [バッチ名] — [バッチタイトル]

### 背景
[なぜこのバッチが必要だったか]

### 議論
[どのような選択肢を検討し、なぜこの方針を採用したか]

### 決定
[実装方針の核心]

### 結果
[実装後の効果、後続バッチへの影響]

### 関連ファイル・コミット
- コミット: [ハッシュ]
- 変更: [ファイルリスト]
```

エントリは時系列順 (末尾追加) で配置する。記載粒度は既存エントリ
(F-1 / F-2 / F-13.B 等) に揃える。

### Task 2: FUTURE_WORK.md の更新

#### 2-A: 完了済みへの移動

- 本バッチで完了した項目を「完了済み」セクションに移動する
- 簡潔な完了記録 (発生バッチ / 対応内容 / 関連ファイル) を残す

#### 2-B: 新規課題の追加

本バッチで発生した新たな将来課題を「緊急度 高 / 中 / 低」のいずれかに
追加する。形式は以下:

```markdown
- **[タイトル]** (発生バッチ)
  - 背景: なぜこれが必要か
  - 対応案: どう対応するか
  - 検討時期: いつ判断するか
  - 関連ファイル: 影響を受けるファイル
```

緊急度の判断基準:
- **緊急度 高**: 次のフェーズ (次の数バッチ以内) で必ず対応すべき
- **緊急度 中**: 実運用データ収集後に判断、または別バッチで計画的に対応
- **緊急度 低**: 時間ある時に検討、Phase 完了後の整理対象

### Task 3: 完了レポートに更新内容を明記

各バッチの完了レポートに必ず以下のセクションを含める。

```markdown
## DECISION_LOG.md への追加内容
[追加したエントリの要約]

## FUTURE_WORK.md への変更内容
- 完了済みに移動した項目: [リスト]
- 緊急度 高に追加した項目: [リスト]
- 緊急度 中に追加した項目: [リスト]
- 緊急度 低に追加した項目: [リスト]
```

これらは省略不可。完了レポート提出前に必ず Task 1 / Task 2 を実施し、
その結果を Task 3 として完了レポートに反映すること。

## 不変原則 (Hydrangea Phase A.5-2 で確立)

各バッチでは以下の 5 つの不変原則を遵守する。

- **不変原則 1**: `src/generation/article_writer.py` を一切変更しない
- **不変原則 2**: `src/generation/script_writer.py` を一切変更しない
  - 例外: `configs/prompts/script/` 配下のプロンプトファイルは変更可能
- **不変原則 3**: `src/triage/` の既存ファイルを一切変更しない
  - 例外: 新規ファイルの追加は可能 (例: `jp_coverage_verifier.py`)
- **不変原則 4**: `src/analysis/` を一切変更しない
- **不変原則 5**: 既存テストを破壊しない

各原則の例外は明示的に許可された場合のみ。例外を必要と判断した場合は
作業を停止し、完了レポートで「変更要望」として報告すること。

## バッチプロンプトテンプレートでの参照方法

各バッチプロンプトの末尾に以下のセクションを必ず含める。

```markdown
## 文書更新義務 (Hydrangea Protocol)

本バッチ完了時に docs/BATCH_PROTOCOL.md に従い、以下を必ず実施する:

1. **DECISION_LOG.md** に本バッチの決定エントリを追加 (Task 1)
2. **FUTURE_WORK.md** の完了済み移動 + 新規課題追加 (Task 2)
3. 完了レポートに上記更新内容を明記 (Task 3)

これらは省略不可。完了レポート提出前に必ず実施すること。
```

---

## メンテナンスルール

- 不変原則の追加・変更は本ドキュメントを更新する
- バッチプロトコル自体の変更も DECISION_LOG.md に記録する
- 形骸化を防ぐため、月 1 回のレビュー対象とする
  (`docs/FUTURE_WORK.md` の月次レビュートリガーと同期)
